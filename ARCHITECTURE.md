# 系统架构文档

## 1. 当前阶段：本地 CLI 工具

```text
Excel 文件
  -> download_orders.py
      -> 解析订单号和 Design Link
      -> 调用 gdown 下载 Google Drive 图片
      -> 输出 orders/<订单号>/
```

当前特点：

- 单机运行。
- 使用 Python 虚拟环境管理依赖。
- 不依赖数据库。
- 不提供 Web 页面。
- 优先解决下图交付。

## 2. 过渡阶段：数据库后台

```text
Excel 文件
  -> 导入服务
      -> PostgreSQL
      -> 后台下载任务
      -> 本地文件或对象存储
```

目标：

- 保存订单、明细、设计图、物流单号。
- 保存导入批次和原始行。
- 支持失败重试。
- 支持导出固定模板。

## 3. 未来阶段：云端服务

```text
浏览器
  -> Web 前端
      -> 后端 API
          -> PostgreSQL
          -> 对象存储
          -> 后台任务队列
              -> 下载设计图
              -> 创建物流单
              -> 查询物流轨迹
          -> 定时任务
```

建议组件：

- 前端：React / Next.js。
- 后端：FastAPI 或 NestJS。
- 数据库：PostgreSQL。
- 文件存储：S3 兼容对象存储。
- 后台任务：Celery / RQ / BullMQ。
- 部署：Docker + 云服务器。

## 4. 文件存储策略

当前：

- 本地目录 `orders/<订单号>/`。

未来：

- 原始文件和图片进入对象存储。
- 数据库只保存文件元数据和访问地址。
- 支持按订单、明细、履约分组查询附件。

## 5. 后台任务策略

适合后台任务的操作：

- Google Drive 图片下载。
- 物流单号创建。
- 物流轨迹定时查询。
- 大批量 Excel 导入导出。

后台任务要求：

- 有任务状态。
- 有失败原因。
- 可重试。
- 可审计。

## 6. 待确认问题

- 最终部署在哪个云平台。
- 是否需要多租户或只服务单一业务团队。
- 文件是否必须长期保存。
- 物流 API 的技术协议和认证方式。
